| BR-008 | プロジェクト・進捗管理 | プロジェクト情報の変更・更新プロセス | 高 | 事業検証の進展やピボットに応じて、登録済みの基本情報を安全に書き換え、プロジェクト全体の方向性やターゲット情報を常に最新化できる業務フローを整備する。 |
| BR-009 | プロジェクト・進捗管理 | APIキー（BYOK）の設定と紐付け管理 | 高 | 各プロジェクトでAIツールを利用可能にするため、ユーザーが自身で用意したAI用のAPIキーを登録・検証する。※要確認：業務要件・ビジネスルール（APIキー利用における免責事項・セキュリティ上の責任分解、およびデータ保護方針の整合性）、※要確認：技術要件・技術制約（持ち込みキーの妥当性確認とセキュアな保管・通信方法） |
| BR-010 | プロジェクト・進捗管理 | 起業プロセスのマイルストーン設定と追跡 | 中 | スタートアップの立ち上げに必要な節目（アイデア評価、プロトタイプ、MVP、プレシード期等）を定義し、計画的な事業推進を図る。※要確認：ドメイン知識（大学のインキュベーション推進プログラムで推奨される標準成長ステージの取り込み要否） |
| BR-011 | プロジェクト・進捗管理 | プロジェクト目標・KPIの策定と状況管理 | 中 | 各マイルストーンに紐づく具体的な定量目標（KPI）を設定し、それに対する進捗状況や検証結果を定期的に記録・評価する仕組みを構築する。 |
| BR-012 | プロジェクト・進捗管理 | 変更履歴の自動記録と世代管理 | 中 | AIによる自動生成コンテンツや、手動での編集内容について、いつ、誰が、どの部分を変更したかを履歴として自動保存し、過去の状態と比較・復元できるようにする。※要確認：業務要件・ビジネスルール（過去の履歴データを保持する期間、および世代数の最大上限ルール） |
| BR-013 | プロジェクト・進捗管理 | プロジェクト全体の進捗ステータス可視化 | 高 | プロジェクト単位で現在どのフェーズ（構想中、検証中、実装中、完了など）に位置しているかをステータスとして管理し、ダッシュボード等で直感的に進捗把握が行えるようにする。 |
| BR-014 | プロジェクト・進捗管理 | プロジェクトのアーカイブおよび安全な削除 | 低 | 活動休止または不要になったプロジェクトについて、誤消去を防ぐためにアーカイブ（参照専用）状態へと移行させ、必要に応じて安全に削除・クリーンアップできる手順を定義する。※要確認：議事録・会議メモ（削除プロセスにおけるデータ復元不可の警告表示、および同意確認フローに関する打ち合わせ合意事項） |
| BR-015 | アイデア検証管理 | 新規スタートアップアイデアの登録・編集プロセス | 高 | 起業学生が新規の起業アイデア（名称、ターゲット顧客、抱える課題、解決策など）を入力・編集し、プロジェクトに紐づけて保存できること。※要確認（業務要件・ビジネスルール：1プロジェクトに対して登録可能なアイデアの上限数および必須入力項目の定義） |
| BR-016 | アイデア検証管理 | AIによる実現可能性（Feasibility）の自動評価プロセス | 高 | 登録されたアイデアを基に、AIが技術的な実現性や参入障壁、想定されるリスクについて多角的に分析し、評価結果をユーザーに提示すること。※要確認（ドメイン知識：スタートアップにおける一般的なアイデア実現可能性の評価基準・フレームワークの適用） |
| BR-017 | アイデア検証管理 | AIによる新規性・独自性（Innovation）の自動評価プロセス | 高 | 入力されたアイデアを基に、既存の類似サービスや競合他社と比較した際の独自性や革新的な要素について、AIが分析および評価を実行すること。 |
| BR-018 | アイデア検証管理 | AIによる市場性（Market Potential）の自動評価プロセス | 中 | ターゲットとなる市場規模や想定顧客の購買意欲の強さ、市場トレンドとの適合性について、AIが客観的な評価と示唆を出力すること。※要確認（技術要件・技術制約：AIが市場トレンド分析に利用する参照データソースの制限や最新情報の取得方法） |
| BR-019 | アイデア検証管理 | AIによるブラッシュアップ・改善提案の取得プロセス | 高 | 各評価（実現可能性・新規性・市場性）の結果を踏まえ、アイデアの弱点を補強するための具体的な改善アクションプランや、次に踏むべきアクションをAIから取得すること。 |
| BR-020 | アイデア検証管理 | AI評価結果の手動編集および更新プロセス | 中 | AIが生成した評価結果や改善提案のテキストをユーザーが自由に手動編集・追記し、確定版として保存できること。また、再生成を行った際にも過去の手動編集内容を履歴として残せること。※要確認（業務要件・ビジネスルール：手動編集データとAI再生成データの競合が発生した際の適用優先ルール） |
| BR-021 | キャンバス分析管理 | キャンバス作成方式の選択プロセス | 高 | ユーザーが対象のスタートアッププロジェクトを選択し、ビジネスの検証フェーズに応じて「Lean Canvas」または「Business Model Canvas」のいずれかを選択して新規作成プロセスを開始できること。 |
| BR-022 | キャンバス分析管理 | AIによるLean Canvas自動生成プロセス | 高 | プロジェクトに登録された基本アイデアを基に、AIがLean Canvasの9つの構成要素（課題、解決策、独自の価値提案、圧倒的優位性、顧客セグメント、主要指標、チャネル、コスト構造、収益の流れ）の初期ドラフトを自動生成するプロセス。※要確認：各要素の解釈基準（ドメイン知識）、AIの生成ロジックおよび制限（技術要件・技術制約） |
| BR-023 | キャンバス分析管理 | AIによるBusiness Model Canvas自動生成プロセス | 中 | プロジェクト情報を基に、AIがBusiness Model Canvasの9つの構成要素（顧客セグメント、価値提案、チャネル、顧客との関係、収益の流れ、主要リソース、主要活動、主要パートナー、コスト構造）を自動生成するプロセス。※要確認：標準フレームワークの定義（ドメイン知識）、生成用プロンプトの管理（技術要件・技術制約） |
| BR-024 | キャンバス分析管理 | キャンバス構成要素の個別編集プロセス | 高 | AIが自動生成したキャンバスの各構成要素（カード単位）に対して、ユーザーが仮説検証や顧客インタビューの結果に基づき、手動でテキストを追加・修正・削除できるプロセス。 |
| BR-025 | キャンバス分析管理 | キャンバスのバージョン保存・世代管理プロセス | 中 | キャンバスを更新する際、変更内容を上書き保存するだけでなく、特定の検討フェーズとして名前を付けて新規バージョンとして保存（世代管理）できるプロセス。※要確認：1プロジェクトあたりの最大バージョン保持制限（業務要件・ビジネスルール） |
| BR-026 | キャンバス分析管理 | 過去バージョンの比較参照プロセス | 中 | ユーザーが現在編集中のキャンバスと、過去に保存した任意のバージョンを画面上で並行して表示し、ビジネスモデルがどのようにピボット（転換）したかを比較・確認できるプロセス。 |
| BR-027 | キャンバス分析管理 | 過去バージョンからの復元プロセス | 中 | 検証の差し戻しや方針転換に伴い、過去の特定のキャンバスバージョンを選択し、現在の最新キャンバスとして強制的に復元（ロールバック）するプロセス。※要確認：復元実行時における現行データのバックアップ自動生成ルール（業務要件・ビジネスルール） |
| BR-028 | キャンバス分析管理 | キャンバス間データ引継ぎ連携プロセス | 低 | 検証初期に作成した「Lean Canvas」のデータ（顧客セグメント、提供価値、チャネル等）を、事業成長に伴い「Business Model Canvas」へ移行する際、共通する項目データを引き継いで新規作成できるプロセス。※要確認：2つのキャンバス間での項目マッピングとデータ変換ルール（ドメイン知識） |
| BR-029 | キャンバス分析管理 | キャンバスのステータス管理プロセス | 中 | 作成したキャンバスに対して「ドラフト」「顧客検証中」「仮説立証済」「アーカイブ」などのライフサイクルステータスを手動で設定・管理するプロセス。※要確認：ステータス変更時の他モジュール（進捗管理）への通知連動（業務要件・ビジネスルール） |
| BR-030 | キャンバス分析管理 | 変更履歴の自動記録プロセス | 低 | 共同編集メンバーやAIによってキャンバスの要素が更新された際、「誰が」「いつ」「どの要素を」変更したかの履歴（監査ログ）を自動でバックグラウンドに記録するプロセス。 |
| BR-031 | SWOT分析管理 | SWOT分析プロセスの開始とコンテキスト情報の連携 | 高 | ユーザーはプロジェクトのビジネスアイデアや検証結果をインプットとして、SWOT分析プロセスを開始する。その際、先行するLean Canvas等の登録情報（※要確認：ドメイン知識）をトリガーとして自動連携し、分析の準備を整える。 |
| BR-032 | SWOT分析管理 | AIを活用した環境要因（内部・外部）の自動抽出業務 | 高 | システムは連携されたプロジェクト情報を基に、内部環境（強み・弱み）および外部環境（機会・脅威）をAIにより自動的に抽出・整理する。ユーザーはこれらを確認し、アイデアの客観的な市場適合性を評価する初期段階の業務とする。 |
| BR-033 | SWOT分析管理 | SWOT要因の手動カスタマイズおよび履歴管理プロセス | 中 | ユーザーはAIが抽出した環境要因を検証し、当事者ならではの知見や追加調査に基づき、手動で各要素を追加・修正・削除する。編集された内容はプロジェクトデータに保存され、後から過去のバージョンと比較参照できるものとする（※要確認：業務要件・ビジネスルール：編集履歴の保持期間と版管理ポリシー）。 |
| BR-034 | SWOT分析管理 | クロスSWOTによる戦略オプションの策定業務 | 高 | ユーザーは確定したSWOT要素を掛け合わせ、強み×機会（積極策）、弱み×脅威（回避策）などのクロスSWOT分析（※要確認：ドメイン知識）を行い、AIから実践的な戦略提案を取得する。この提案を基に、進むべき事業の方向性と優先度を決定する業務プロセスとする。 |
| BR-035 | SWOT分析管理 | 分析結果の共有およびメンターレビュープロセス | 中 | ユーザーは策定したSWOT分析結果と戦略提案を、PDF等でエクスポートして共有する、またはプラットフォーム上で共同創業者やメンターに公開してレビューを依頼する（※要確認：議事録・会議メモ：メンターレビュー時のコメント入力と修正指示のフィードバック連携プロセス）。 |
| BR-036 | MVPロードマップ管理 | MVP対象機能の選定およびMoSCoW優先度設定プロセス | 高 | ユーザーは、ビジネスアイデア実現に必要な機能一覧を抽出し、MoSCoW分析手法を用いてMVP（最小実用プロダクト）に含めるコア機能を特定・分類する。※要確認：各優先度（Must/Should/Could/Won't）を定義・判定するための客観的なビジネスルールと判断基準。 |
| BR-037 | MVPロードマップ管理 | プロダクトバックログの構築・詳細化プロセス | 高 | ユーザーは、MVP対象として選定した機能を開発者が実装可能な単位（ユーザーストーリーやタスク）に分解し、受け入れ基準を明確にしたプロダクトバックログとして構造化・整理する。 |
| BR-038 | MVPロードマップ管理 | AI支援によるMVPロードマップの自動生成および合意プロセス | 高 | ユーザーは、定義したバックログおよび制約条件（目標リリース時期等）を基に、AIが提案する開発フェーズ分けとロードマップ案を確認・編集し、プロジェクトの初期ロードマップとして確定させる。 |
| BR-039 | MVPロードマップ管理 | 開発規模（工数・期間）の初期見積り策定プロセス | 中 | ユーザーは、各バックログ項目の実装難易度や開発リソース（開発メンバーのスキルや人数）を考慮し、開発工数（人日またはストーリーポイント）の初期見積りを行う。※要確認：スタートアップ開発における初期ベロシティの目安や見積り基準に関するドメイン知識。 |
| BR-040 | MVPロードマップ管理 | 開発マイルストーン設定および進捗モニタリングプロセス | 中 | ユーザーは、MVPリリースに向けた重要マイルストーン（アルファ版、ベータ版、リリース等）を設定し、開発タスクの進捗状況とロードマップ上の計画とのギャップを定期的に確認・評価する。 |
| BR-041 | MVPロードマップ管理 | ロードマップおよびバックログの変更管理（ピボット対応）プロセス | 中 | ユーザーは、顧客検証や開発状況の変化に応じて計画を変更（ピボット）する際、変更前の計画を履歴（バージョン）として保存し、変更理由を記録する。※要確認：共同創業者やステークホルダーへの変更通知・合意形成に関するビジネスルール。 |
| BR-042 | MVPロードマップ管理 | メンターレビューおよびアドバイス反映プロセス | 低 | ユーザーは、策定したMVPロードマップを大学のメンターや専門家に提示してレビューを受け、得られたフィードバックをロードマップに反映・調整する。※要確認：外部アドバイザリーミーティングでの合意事項（議事録・会議メモ）をロードマップの修正履歴に紐付けるプロセス。 |
